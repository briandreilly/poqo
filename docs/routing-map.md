# Routing Map

This map is frozen unless a repeated failure proves a boundary is wrong. Prefer boundary fixes over new classes.

## DIRECT Classes

- direct question: short, explicit, question-shaped, and answer-ready
- direct utterance: greeting, observation, simple reaction, non-actionable remark
- personally scoped evaluation: bounded by the speaker's own experience
- safe simple subjective prompt: answerable without proof or clarification

## NARROW Classes

- vague concern: a concern is signaled, but what feels wrong is still missing
- unsupported stance: strong claim without stated basis
- contested framing: multiple linked definitional or ideological claims without a shared frame
- coherent critique bundle: a clear critique where the next useful move is to ask what should change
- underspecified decision: missing criteria, missing source, or conflicting constraints
- argument-loaded statement: strong multi-claim statement that is not a clean direct-answer prompt

## PROVE Boundary

PROVE is allowed only when the claim is sufficiently defined and the answer materially stands on visible support.

- `document`: the answer depends on provided text
- `chat`: the answer depends on prior stated constraints or active chat context
- `world`: the answer depends on outside facts that can change

If the prompt is contested, unsupported, or frame-misaligned, poqo should NARROW before PROVE.

## Support Layers

- domain lock is a support layer, not a routing class
- intervention mode is a posture layer, not a routing class

## Change Discipline

- repeated failure before route-map change
- boundary fix before category growth
- smallest local fix before new abstraction
